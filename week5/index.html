    • Debugging

  • there is always the alert() method to show a dialog during certain points of code. It stops the program from running until OK is clicked so we can put breakpoints in the code.
  • Another way of debugging is using the console. console.log() will output anything we tell it to. console.trace() will show the stack trace of the functions that were called in the lead up to that exception
  • putting in the word "debugger;" in your code will cause a breakpoint in the browser and pause the code while you can hover over variables and check their values. should remember to remove all of those lines before the final product is released
  • there are a number of error objectst that can be used for specific errors
  • JS throws exceptions when it finds errors but we can manually invoke an excep[tion but the 'throw' statement. If something is thrown the code will stop executing it is best practice to throw an error object.

    • Exception Handling

  • During development, when a program throws an exception we can go and fix the exception. In production however we do not want the program to break. This is fixed through execption handling. It will had any errors from the users but still identify that there is an error.
  • If a piece of code may end in an exception we put it inside a try, catch. The try will run the code like normal and skip the catch if everything works. If there is an error then the catch will execute other code. There is also a 'finally' that will be ran no matter what.
